# KF-002: Add CLI command to set fleet orchestration turn limit

## PURPOSE

Operators currently can't tune orchestration turn limits without editing config or code. Expose a CLI command so the limit can be changed quickly per-environment without redeploying.

## CURRENT_STATE

There is no single "fleet orchestration turn limit" today — multiple, partially overlapping turn-cap knobs exist:

- **`worker_max_turns`** (per-agent, identity.yaml) — outer worker continuation loop. After each turn that ends with `STATUS: IN_PROGRESS`, the worker re-prompts up to this many times within a single run. Default 5. Read at `packages/cli/src/orchestration/worker-heartbeat.ts:211` (`getIdentityForRoot(root).worker_max_turns ?? 5`). Type declared at `packages/cli/src/types.ts:138`. No setter helper exists; today operators hand-edit identity.yaml.
- **`heartbeat_max_inner_turns`** (per-agent, identity.yaml) — inner Claude SDK turn cap within one heartbeat call. Default 50. Resolver: `getHeartbeatMaxInnerTurnsForRoot()` in `packages/cli/src/config.ts:164`. Used by `worker-heartbeat.ts:212` and `ceo-heartbeat.ts:361`.
- **Per-agent `maxTurns`** in `agents/<name>/agent.yaml` (sub-agent definition). Default 10. Already operator-tunable via `kyberbot agent create --max-turns` (`packages/cli/src/commands/agent.ts:80`) and shown by `kyberbot agent show` (`agent.ts:150`). Not what the job is about.
- **Hardcoded values**: 30 (telegram, whatsapp), 15 (heartbeat default chat), 1 (brain extractors).
- **Dispatcher (delamain) `maxTurns`** in `.als/modules/.../runtime-manifest.config.json` — separate ALS subsystem, not part of "fleet orchestration."

Persistent settings infrastructure that exists today:
- **`orchestration_settings` SQLite singleton table** at `packages/cli/src/orchestration/db.ts:278`. Columns: `orchestration_enabled`, `heartbeat_interval`, `active_hours_start`, `active_hours_end`. Reader/writer: `packages/cli/src/orchestration/settings.ts`. HTTP endpoints: `GET/PUT /fleet/orch/settings` at `packages/cli/src/server/orchestration-api.ts:689,693`. **No turn-limit field today.** Adding one would require: (1) a column + migration, (2) a field on `OrchestrationSettings` (`packages/cli/src/orchestration/types.ts:205`), (3) wiring into `getOrchestrationSettings/updateOrchestrationSettings`, (4) a consumer change in worker-heartbeat to read from DB.
- **Fleet registry** (`~/.kyberbot/registry.yaml`) — used by `kyberbot fleet defaults --auto-start` (`fleet.ts:517`) but is a fleet-listing/registry concern, not a runtime-config one.

CLI surface today:
- `kyberbot fleet ...` (`packages/cli/src/commands/fleet.ts`): `list`, `register`, `register-remote`, `unregister`, `start`, `stop`, `status`, `defaults`. The closest precedent for a setter is `fleet defaults --auto-start <names>` (registry mutation).
- `kyberbot orch ...` (`packages/cli/src/commands/orch.ts`): wraps `/fleet/orch/*` HTTP endpoints — goals, issues, org, inbox, activity, init, status. Currently has **no** `settings` subcommand even though the HTTP endpoint exists. This is the more semantically natural home for orchestration runtime knobs.

## REQUIREMENTS

Re-targeted after operator answers to the 4 research questions (2026-05-06).

- **Command shape**: `kyberbot fleet set-turns <N>` (writer) and `kyberbot fleet get-turns` (reader). Help text must state explicitly that this tunes `heartbeat_max_inner_turns` (the inner Claude SDK turn cap, default 50) and does NOT touch `worker_max_turns` or per-agent sub-agent `maxTurns`.
- **Field mutated**: `heartbeat_max_inner_turns` in `<agent_root>/identity.yaml`. No schema change anywhere else. No SQLite migration. No registry change.
- **Validation**: N must be a positive integer (≥1). Match the resolver's existing guardrail at `packages/cli/src/config.ts:167` (`Number.isFinite(raw) && raw >= 1`). Reject non-numeric, zero, negative, fractional with a clear stderr message and non-zero exit.
- **Agent targeting**: accept optional `--agent <name>` flag. When supplied, resolve the agent root via `resolveAgentRoot()` at `packages/cli/src/registry.ts:122` (handles both registry name and path-like inputs). When omitted, default to CWD via `getRoot()`. Reject remote agents (registry entries with `type: 'remote'` / no `root`) with a clear error — identity.yaml mutation requires local filesystem access.
- **Persistence + propagation**: `getIdentityForRoot` caches per root (`packages/cli/src/config.ts:304`), but `worker-heartbeat.ts:211` and `ceo-heartbeat.ts:357` re-import `config.js` and call the resolver per run, and there's already an `identity-watcher` plus `clearIdentityCache(root)` helper at `config.ts:333`. The setter MUST call `clearIdentityCache(root)` after writing so the next heartbeat in the same process sees the new value (no restart required).
- **`get-turns` semantics**: print the *effective* value (resolver result from `getHeartbeatMaxInnerTurnsForRoot(root)`), with a note when the value is the default-50 fallback vs. an explicit identity.yaml setting.
- **Output**: setter prints `PRIMARY` `Set inner turn limit to N for agent <name>` plus a `DIM` `Takes effect on next heartbeat.` line — mirrors the `kyberbot bus subscribe` output convention at `commands/bus.ts:389-390`.
- **Help text disambiguation** (mandatory): `--help` for `set-turns` and `get-turns` must list (a) the exact identity.yaml field, (b) the default (50), (c) the two consumers (worker heartbeat, CEO heartbeat), and (d) what is NOT controlled (worker continuation loop `worker_max_turns`, sub-agent `maxTurns` — that one has its own setter via `kyberbot agent create --max-turns`).

## RESEARCH

**Feasibility: high.** Re-target is a write-only delta on top of an existing read path. The resolver, identity.yaml, the cache-invalidation helper, and an established read-mutate-write idiom for identity.yaml all exist. Estimated change: ~80 lines in `packages/cli/src/commands/fleet.ts` plus a unit test.

**Re-targeted read site (verified):** `getHeartbeatMaxInnerTurnsForRoot()` at `packages/cli/src/config.ts:164-170` reads `heartbeat_max_inner_turns` from identity.yaml, validates `Number.isFinite(raw) && raw >= 1`, and falls back to 50. Field is declared optional `number` at `packages/cli/src/types.ts:54`. Two consumers: `packages/cli/src/orchestration/worker-heartbeat.ts:212` (`maxInnerTurns`) and `packages/cli/src/orchestration/ceo-heartbeat.ts:357,361` (passed as `maxTurns` to `client.complete`). Both re-resolve via dynamic `import('../config.js')` per run.

**Mutation idiom (verified pattern to mirror):** `packages/cli/src/commands/bus.ts:370-388` (subscribe) and `commands/bus.ts:407-423` (unsubscribe) — `readFileSync` → `yaml.load as Record<string, unknown>` → mutate → `writeFileSync(path, yaml.dump(identity, { lineWidth: 120 }), 'utf-8')`. Same pattern in `commands/channel.ts:139,157,178`, `commands/onboard.ts:226`, `commands/update.ts:335`, and `templates/auto-migrate.ts:71`. Lock-step with this convention; no need to introduce a new helper module.

**Cache invalidation (verified):** `clearIdentityCache(root)` at `packages/cli/src/config.ts:333` exists and is exported. Must be called after the write so a same-process heartbeat picks up the new value without a restart. The `identity-watcher` (referenced by `runtime/identity-watcher.test.ts`) also fires on filesystem changes, so any out-of-process consumers (running fleet servers) will refresh on their next reload tick.

**Agent-root resolution (verified):** `resolveAgentRoot(nameOrPath)` at `packages/cli/src/registry.ts:122` already handles both name lookup (via `~/.kyberbot/registry.yaml`) and path inputs (`./`, `~/`, `/`, `.`). Throws with a clear message if the name is not registered. Remote agents have `type: 'remote'` and no `root` field (`registry.ts:212-219`); the new command must check this explicitly and bail before attempting to read identity.yaml.

**Identified risks / footguns:**
1. **Naming collision (still applies, now critical).** Three turn-counters exist — `worker_max_turns` (default 5), `heartbeat_max_inner_turns` (default 50, this command's target), sub-agent `maxTurns` (default 10). Help text must spell out the field name and explicitly disclaim the other two, otherwise operators will assume a "turn limit" CLI controls all three.
2. **"Fleet" wording is a slight mislabel.** The command writes a per-agent identity.yaml field, scoped to the agent at CWD or `--agent <name>`. There is no fleet-wide override. Help text should say "for the targeted agent."
3. **Hardcoded turn caps elsewhere.** `maxTurns: 30` in `server/channels/telegram.ts:130` and `whatsapp.ts:88`, and `maxTurns: 1` in brain extractors are unrelated and out of scope. Help text should not promise to affect them.
4. **Cache invalidation must not be skipped.** If the setter forgets `clearIdentityCache(root)`, an in-process heartbeat will continue to use the old cached identity until process restart — making the "takes effect on next heartbeat" promise false in long-running fleets.
5. **Concurrent writes.** identity.yaml is also mutated by `bus subscribe/unsubscribe`, `channel` setters, `onboard`, and `update`. No file locking exists today; concurrent CLI invocations could interleave. Out of scope for this job (matches the rest of the codebase's posture), but worth noting.

**Patterns to lean on:**
- `kyberbot bus subscribe` (`commands/bus.ts:370-394`) — read/mutate/write identity.yaml + "takes effect on next start" output line.
- `kyberbot agent create --max-turns` (`commands/agent.ts:80`) — `parseInt` argument parsing precedent.
- `kyberbot fleet defaults` (`commands/fleet.ts:517-545`) — same-subcommand "set when arg supplied, show when not" branch as a backup pattern (we are doing two separate subcommands per operator's Q3, but this is a fallback if we ever consolidate).
- `resolveAgentRoot` (`registry.ts:122`) — agent-name → root resolution for `--agent`.
- `clearIdentityCache` (`config.ts:333`) — same-process cache invalidation after the write.

**Out of scope (confirmed):** dispatcher `maxTurns` in `runtime-manifest.config.json` (ALS-side, separate subsystem); sub-agent `maxTurns` (already has setter via `agent create --max-turns`); `worker_max_turns` (operator deferred via Q1 — can be a follow-up job); hardcoded channel handler turn caps; SQLite `orchestration_settings` schema changes (operator chose Q2 → A, stay in identity.yaml); `kyberbot orch settings` mirror command (operator chose Q3 → A; can be a follow-up job).
