# KF-002: Add CLI command to set fleet orchestration turn limit

## PURPOSE

Operators currently can't tune orchestration turn limits without editing config or code. Expose a CLI command so the limit can be changed quickly per-environment without redeploying.

## CURRENT_STATE

There is no single "fleet orchestration turn limit" today — multiple, partially overlapping turn-cap knobs exist:

- **`worker_max_turns`** (per-agent, identity.yaml) — outer worker continuation loop. After each turn that ends with `STATUS: IN_PROGRESS`, the worker re-prompts up to this many times within a single run. Default 5. Read at `packages/cli/src/orchestration/worker-heartbeat.ts:211` (`getIdentityForRoot(root).worker_max_turns ?? 5`). Type declared at `packages/cli/src/types.ts:138`. No setter helper exists; today operators hand-edit identity.yaml.
- **`heartbeat_max_inner_turns`** (per-agent, identity.yaml) — inner Claude SDK turn cap within one heartbeat call. Default 50. Resolver: `getHeartbeatMaxInnerTurnsForRoot()` in `packages/cli/src/config.ts:164`. Used by `worker-heartbeat.ts:212` and `ceo-heartbeat.ts:361`.
- **Per-agent `maxTurns`** in `agents/<name>/agent.yaml` (sub-agent definition). Default 10. Already operator-tunable via `kyberbot agent create --max-turns` (`packages/cli/src/commands/agent.ts:80`) and shown by `kyberbot agent show` (`agent.ts:150`). Not what the job is about.
- **Hardcoded values**: 30 (telegram, whatsapp), 15 (heartbeat default chat), 1 (brain extractors).
- **Dispatcher (delamain) `maxTurns`** in `.als/modules/.../runtime-manifest.config.json` — separate ALS subsystem, not part of "fleet orchestration."

Persistent settings infrastructure that exists today:
- **`orchestration_settings` SQLite singleton table** at `packages/cli/src/orchestration/db.ts:278`. Columns: `orchestration_enabled`, `heartbeat_interval`, `active_hours_start`, `active_hours_end`. Reader/writer: `packages/cli/src/orchestration/settings.ts`. HTTP endpoints: `GET/PUT /fleet/orch/settings` at `packages/cli/src/server/orchestration-api.ts:689,693`. **No turn-limit field today.** Adding one would require: (1) a column + migration, (2) a field on `OrchestrationSettings` (`packages/cli/src/orchestration/types.ts:205`), (3) wiring into `getOrchestrationSettings/updateOrchestrationSettings`, (4) a consumer change in worker-heartbeat to read from DB.
- **Fleet registry** (`~/.kyberbot/registry.yaml`) — used by `kyberbot fleet defaults --auto-start` (`fleet.ts:517`) but is a fleet-listing/registry concern, not a runtime-config one.

CLI surface today:
- `kyberbot fleet ...` (`packages/cli/src/commands/fleet.ts`): `list`, `register`, `register-remote`, `unregister`, `start`, `stop`, `status`, `defaults`. The closest precedent for a setter is `fleet defaults --auto-start <names>` (registry mutation).
- `kyberbot orch ...` (`packages/cli/src/commands/orch.ts`): wraps `/fleet/orch/*` HTTP endpoints — goals, issues, org, inbox, activity, init, status. Currently has **no** `settings` subcommand even though the HTTP endpoint exists. This is the more semantically natural home for orchestration runtime knobs.

## REQUIREMENTS

Carrying forward the operator's stated requirements with research-derived clarifications:

- New CLI subcommand to set the persisted limit. Provisional name `kyberbot fleet set-turns <N>` per the description, but see RESEARCH_QUESTIONS Q3 — `kyberbot orch settings turns <N>` may be a better fit.
- Companion read command `kyberbot fleet get-turns` (or `orch settings show`).
- Validation: N is a positive integer (≥1). Reject non-numeric, zero, negative, non-integer with a clear stderr message and non-zero exit.
- Persistence across restart — destination depends on Q2.
- Effect on next run without restart: feasible for both candidate destinations. identity.yaml is re-read on every heartbeat (`getIdentityForRoot` is called fresh in `worker-heartbeat.ts:211`); the SQLite singleton is also read per-run (`heartbeat.ts:141`). Either works without restart.
- `--help` output and docs updated. No existing doc location for fleet runtime knobs has been identified — likely just inline `commander` `.description()` is sufficient.

## RESEARCH

**Feasibility: high.** The infrastructure to land this is already in place; the change is small (add a setter/getter pair on top of existing config). The only real decision is which of the existing knobs this command controls and where it persists. No architectural scaffolding is needed.

**Identified risks / footguns:**
1. Naming collision — calling the field "turns" is ambiguous because three different turn-counters exist (`worker_max_turns`, `heartbeat_max_inner_turns`, sub-agent `maxTurns`). Whichever the command writes, the help text needs to be explicit so an operator does not assume it controls all three.
2. Per-agent vs fleet-wide — the description says "fleet" but every existing turn-cap is per-agent. There is no fleet-wide override mechanism today; introducing one would either require a registry-level field with a new fall-through resolver, or simply documenting that "fleet set-turns" run from an agent root mutates that agent's identity.
3. Hardcoded `maxTurns: 30` in channel handlers (`telegram.ts:130`, `whatsapp.ts:88`) is unrelated to orchestration but operators may expect a "turn limit" CLI to control it. Out of scope; flag in help text.
4. Effective scope — `worker_max_turns` defaults to 5, but `heartbeat_max_inner_turns` defaults to 50. The job's phrasing "maximum orchestration turns per task" matches `worker_max_turns` more closely (it is literally the per-task continuation cap), but operators raising the value high (e.g. 100) might mean the inner cap. Help text should disambiguate.

**Patterns to lean on:**
- `kyberbot agent create --max-turns` (`commands/agent.ts:80`) for argument parsing with `parseInt`.
- `kyberbot fleet defaults --auto-start` (`commands/fleet.ts:517`) for "set / show" pattern in a single subcommand.
- `kyberbot orch ...` HTTP-wrapper pattern (`commands/orch.ts:42`) if we route through `/fleet/orch/settings`.
- Existing `OrchestrationSettings` resolver/writer pair (`orchestration/settings.ts`) if we choose the SQLite path.

**Out of scope (confirmed):** dispatcher `maxTurns` in `runtime-manifest.config.json` (ALS-side, separate subsystem); per-agent `maxTurns` (already has CLI); hardcoded channel handler values.

## RESEARCH_QUESTIONS

The phrase "fleet orchestration turn limit" maps to multiple existing knobs and possible storage locations. Operator decisions are needed before planning.

**Q1. Which turn limit does this command control?**

- **Option A — `worker_max_turns`** (outer worker continuation loop, currently default 5, identity.yaml). This is the closest semantic match to "orchestration turns the fleet will run per task" — it caps how many continuation passes a worker takes on a single issue.
- **Option B — `heartbeat_max_inner_turns`** (inner Claude SDK turn cap, currently default 50, identity.yaml). Caps tool-use rounds inside a single Claude call. Less likely the intent, but operators sometimes conflate the two.
- **Option C — Both, with two separate subcommands** (e.g. `set-turns --outer N` / `set-turns --inner N`).
- **Option D — A new fleet-wide ceiling** that overrides both per-agent values, stored in `orchestration_settings` SQLite singleton.

**Recommendation: A.** It matches the description ("turns per task"), is the cap most operators reach first (default is only 5), and has no setter today. B can be a follow-up; C/D add scope without clear demand.

**Q2. Where does the value persist?**

- **Option A — identity.yaml (per-agent)**, mutating the existing `worker_max_turns` field. Pros: matches the existing source-of-truth, no schema changes, immediate effect (re-read each heartbeat at `worker-heartbeat.ts:211`). Cons: the command is named `fleet set-turns` but only mutates the current agent's config — semantically per-agent, not per-fleet.
- **Option B — `orchestration_settings` SQLite singleton**, adding a `worker_max_turns` column + migration. Pros: actually a fleet-singleton feel; existing CRUD via `getOrchestrationSettings`/`updateOrchestrationSettings`. Cons: requires a new resolver in worker-heartbeat to prefer DB value over identity.yaml, schema change, migration, and creates two sources of truth for the same number.
- **Option C — fleet registry (`~/.kyberbot/registry.yaml`)**, adding a `defaults.worker_max_turns`. Pros: genuinely fleet-wide (spans all registered agents). Cons: a new resolver pathway is needed; identity.yaml currently wins.

**Recommendation: A.** Smallest delta, matches existing source-of-truth, no schema change. The "fleet" wording in the command is a minor mislabel that the help text can clarify ("sets `worker_max_turns` for the agent rooted at CWD").

**Q3. Where does the command live in the CLI tree?**

- **Option A — `kyberbot fleet set-turns <N>` / `kyberbot fleet get-turns`** (literally what the description says). Pros: matches operator's stated phrasing. Cons: `fleet` today is about registering/starting/stopping agents — runtime knobs feel out of place.
- **Option B — `kyberbot orch settings turns <N>`** (and `orch settings show`). Pros: `orch` already wraps the orchestration HTTP API; `/fleet/orch/settings` already exists. Most semantically correct.
- **Option C — Both as aliases** — `fleet set-turns` re-routes to `orch settings turns`.

**Recommendation: A** (per the explicit description). Easy to add a `kyberbot orch settings` mirror later if Q2 ends up being B/C.

**Q4. Should the command accept an explicit agent name (multi-agent fleets)?**

`kyberbot fleet set-turns <N> [--agent <name>]` would let an operator tune another registered agent without `cd`-ing into its root. Without `--agent`, the command operates on the agent at CWD (matches every other agent-mutating command).

**Recommendation: ship without `--agent` in v1**; add later if multi-agent operators ask. Reduces scope.

### Operator Answers (2026-05-06)

- **Q1 → B: `heartbeat_max_inner_turns`.** The command controls the inner Claude SDK turn cap (default 50, in identity.yaml), NOT `worker_max_turns`. Update RESEARCH and REQUIREMENTS to target this field instead. Re-check the read site at `packages/cli/src/config.ts:164` (`getHeartbeatMaxInnerTurnsForRoot()`) and consumers `worker-heartbeat.ts:212`, `ceo-heartbeat.ts:361`.
- **Q2 → A: identity.yaml per-agent.** Mutate the existing field directly; no schema change.
- **Q3 → A: `kyberbot fleet set-turns <N>` / `kyberbot fleet get-turns`.** Matches stated phrasing. Help text must clarify it tunes `heartbeat_max_inner_turns` specifically.
- **Q4 → B: include `--agent <name>` in v1.** Operator can target another registered agent without cd-ing into its root. Resolve agent root via the registry (`~/.kyberbot/registry.yaml`) when `--agent` is supplied; default to CWD otherwise.
