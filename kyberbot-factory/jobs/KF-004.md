# KF-004: Telegram Improvement

## PURPOSE

Today, when an operator sends a Telegram message that triggers a longer-running orchestration task, the bot goes silent while it works — no acknowledgement, no typing indicator, no signal that the message was received and is being acted on. This produces a confusing UX: the operator can't tell whether the message was lost, the bot is broken, or work is genuinely in progress. Surface those two missing affordances so the Telegram channel feels responsive and trustworthy.

## CURRENT_STATE

The Telegram adapter lives at `packages/cli/src/server/channels/telegram.ts`. It uses `grammy` v1.30 (`Bot`, `bot.api`). Tests are at `packages/cli/src/server/channels/telegram.test.ts` (vitest, fully mocked grammy + claude client).

Owner-message routing (the only path that triggers long work) is in `bot.on('message:text', …)` at `telegram.ts:62–165`. After the verification + owner-guard + `/start` branches, the handler:

1. Builds a `ChannelMessage` (lines 109–119).
2. If `this.messageHandler` is set (via `onMessage()`), delegates to it (lines 121–123). **A grep across `packages/cli/src` shows no production caller ever calls `channel.onMessage(...)` — only tests do.** So in practice the `else` branch always runs.
3. Otherwise (lines 124–164) calls `getClaudeClient().complete(prompt, { system, maxTurns: 30, subprocess: true, cwd: this.root })`, then `await ctx.reply(reply)` (with chunking past 4096 chars).

`client.complete()` is defined at `packages/cli/src/claude.ts:119` and shells out to the Claude Code CLI (`@anthropic-ai/claude-code` ^2.1.74) as a subprocess with up to 30 turns. This is the long-running path — empirically always many seconds, often tens of seconds. There is **no separate "worker / orchestration loop" abstraction in the Telegram adapter**: every owner text message flows synchronously through `complete()` and then `ctx.reply()`.

There is no current acknowledgement message and no `sendChatAction` / typing-indicator usage anywhere in `packages/cli/src` (verified via grep). The `Channel` interface at `packages/cli/src/server/channels/types.ts` does not expose any "in-progress" hook.

## REQUIREMENTS

### Acknowledgement message for long-running tasks

- When the bot determines a request will be long-running (e.g. anything routed through the worker / orchestration loop, or anything that will take more than ~3 seconds), send a short acknowledgement message to the same Telegram chat *before* starting the work.
- Acknowledgement content: human, terse, and informative. Default suggestion: `Starting on it…` plus a one-line restatement of what's being worked on when that's cheap to compute. Tune wording during planning if the agent finds a better convention in the existing codebase.
- Do NOT send an acknowledgement for short / instant replies — the bot already responds quickly enough that an extra "starting" message would be noise.
- The threshold for "long-running" must be a single, named decision (constant or config) with a comment explaining the choice, not scattered magic numbers.

### Typing indicator while working

- Use the Telegram Bot API `sendChatAction` (`action: "typing"`) to display the typing indicator in the chat for the duration of any long-running task.
- Telegram's typing action auto-expires after ~5 seconds. The implementation must refresh the action on a timer (e.g. every 4 seconds) so the indicator remains continuously visible until the work completes or fails.
- Stop the typing indicator promptly when the task finishes (success or failure) — do not leave a dangling refresh loop after the final reply is sent.
- The typing indicator must be cancellable / cleanup must run even if the task throws, is cancelled, or times out.

### Cross-cutting

- No regression for short / synchronous replies — they continue to respond as fast as they do today, with no acknowledgement and no typing indicator.
- No regression for other channels (Slack, WhatsApp, Discord, etc.) — this work is scoped to the Telegram adapter only.
- Add at least one test for each of the two behaviors (acknowledgement-emitted vs not, typing-indicator started + refreshed + stopped) using whatever Telegram-adapter test pattern already exists in the repo.
- Update inline help / docstrings on any new public function or setting introduced.
- Token / chat ID / network errors from the Telegram API must not crash the orchestration task; they should be logged and the task continues without the affordance rather than failing the user-visible work.

## RESEARCH

### Library support

- `grammy` ^1.30 exposes `bot.api.sendChatAction(chatId, action)` and accepts `'typing'` as the action (Telegram Bot API contract). No new dependency needed.
- Telegram's `typing` action self-clears after ~5s, confirming the requirement that a refresh timer is needed; setInterval ~4000ms is the standard pattern.
- There is no Telegram API for "stop typing" — the indicator is cleared by sending the actual reply, by sending a different action, or by letting the 5s window lapse. So "stop promptly" naturally falls out of clearing the refresh timer just before `ctx.reply(...)` (at which point the reply itself terminates the typing state in the client UI).

### Where to add the affordances

The natural seam is the `try { … } catch { … }` block at `telegram.ts:126–163` — wrap the `client.complete()` call with:

1. An optional pre-call ack `await ctx.reply(...)`.
2. A typing-loop started before `client.complete()` and stopped in a `finally` (so it's cleaned up on success, throw, and the existing catch's `ctx.reply('Sorry, …')`).

The same pattern can be reused if/when `messageHandler` is wired up by an external caller — a small helper (e.g. `withTypingIndicator(ctx, async () => { … })`) would let either branch opt in. The existing handler-delegation branch (lines 121–123) currently has no production caller, so adding the affordance only on the in-adapter Claude path is sufficient for "no regression" while leaving a clean extension point.

### Threshold semantics — important nuance

The job's REQUIREMENTS describe "long-running" as "anything routed through the worker / orchestration loop, or anything that will take more than ~3 seconds" and ask for a single named threshold. In this codebase:

- There is no separate worker / orchestration loop on the Telegram path — every non-`/start` owner message routes through `client.complete()` (Claude Code subprocess, maxTurns 30).
- That call is essentially always >3s in practice.
- We cannot know in advance how long a given `complete()` will take — duration is only knowable after the fact.

Two viable interpretations of the threshold, both compatible with the requirement of "a single named decision":
- **(A) Always-on**: every routed owner message gets the ack + typing loop. The named constant becomes a boolean policy ("treat all Claude-routed messages as long-running"). Simplest; matches reality that every message is long.
- **(B) Optimistic-delay**: schedule the ack message + typing loop to fire after `LONG_RUNNING_DELAY_MS` (e.g. 1500–2000ms). If `complete()` resolves before the timer fires, cancel both and reply normally. The named constant is the delay. Preserves the spirit of "no acknowledgement for short replies", at the cost of more code.

Either is implementable in <50 lines. Choice belongs to the operator — see RESEARCH_QUESTIONS.

### Acknowledgement wording

The job suggests `Starting on it…` plus an optional one-line restatement "when that's cheap to compute." Restating the user's request faithfully without an LLM is hard (truncating raw input is rude; summarizing requires another model call which would itself be long-running and defeat the point). Default of just `Starting on it…` is the only cheap option in this codebase. Tone in the rest of the adapter (`Hey! I'm {agent}.`, `Sorry, I encountered an error processing your message.`) is casual-terse, consistent with that wording.

### Error isolation

Per the requirement that Telegram API errors must not crash the orchestration: both the ack `ctx.reply(...)` and each `bot.api.sendChatAction(...)` call need their own `try { … } catch (err) { logger.warn(...) }` wrappers. The outer task `try/catch` at `telegram.ts:126/160` already swallows top-level errors but currently routes them into a user-visible `Sorry, …` reply — that path must remain reachable, so the typing-loop cleanup must be in a `finally`, not inside the existing catch.

### Test patterns available

`telegram.test.ts` already mocks `grammy.Bot` with a stub that exposes `bot.api.sendMessage`. To test `sendChatAction`, the existing mock at `telegram.test.ts:72–81` needs one extra method on `api` (`sendChatAction: vi.fn()`) and the new tests can assert it was called, refreshed (using `vi.useFakeTimers()` + `vi.advanceTimersByTime(4000)`), and stopped. Vitest fake-timers are the obvious tool; no new test infrastructure required.

### Other channels

`whatsapp.ts` has no analogous behavior today (verified via grep — no `typing` / `sendChatAction` references). The job explicitly scopes this work to Telegram only, so WhatsApp/Slack/Discord are out of scope. The optional `withTypingIndicator` helper, if introduced, should live next to `telegram.ts` (or in `channels/` if generalized later) — not promoted into `Channel` interface in `types.ts`.

### Risk / complexity assessment

Low complexity, ~1 session of work. Single file (`telegram.ts`) plus one test file. No schema/config changes required for option A; option B adds one numeric constant. No cross-package coordination. Main risk is timer leak on the unhappy path (throw before `finally`), which is mitigated by putting `clearInterval` in a `finally` block rather than after `await complete()`.

## RESEARCH_QUESTIONS

### 1. Threshold policy: always-on vs optimistic-delay?

In this codebase every owner Telegram message routes through `client.complete()` (Claude Code subprocess, maxTurns 30), which is essentially always >3 seconds. There is no fast/sync path to exclude. The REQUIREMENTS say "do NOT send an acknowledgement for short / instant replies", but no such replies actually exist on the Telegram path today.

- **Option A — Always-on.** Treat every Claude-routed message as long-running. Send ack immediately, start typing loop immediately, stop on reply/error. Named constant becomes a policy boolean (e.g. `ACK_ALL_CLAUDE_ROUTED_MESSAGES = true`).
- **Option B — Optimistic delay.** Wait `LONG_RUNNING_DELAY_MS` (suggest 1500ms) before firing the ack and starting the typing loop. If `complete()` returns first, cancel both. Preserves the original "no ack for short replies" intent in case the routing landscape changes (e.g. a future fast-path keyword handler).

**Recommendation: Option B with `LONG_RUNNING_DELAY_MS = 1500`.** It matches the wording of the requirement literally, is future-proof against fast-path replies being added later (e.g. a `/help` shortcut), and the extra code is ~10 lines (a `setTimeout` + `clearTimeout`). The cost over Option A is negligible; the cost of choosing A and having to retrofit later is non-trivial.

### 2. Acknowledgement content: fixed string or attempt restatement?

The job says "Default suggestion: `Starting on it…` plus a one-line restatement of what's being worked on when that's cheap to compute." In this codebase, generating a faithful restatement would require an additional LLM call (truncating raw input verbatim risks rudeness/confusion).

- **Option A — Fixed string `Starting on it…`.** Cheap, always works, consistent.
- **Option B — Echo the first ~80 chars of the user message.** "Starting on it… (re: '{first 80 chars}…')". Free, but reads oddly when the user message is already short, and adds quoting noise.
- **Option C — One extra fast LLM call (e.g. Haiku) for a one-line summary.** Defeats the latency goal; not recommended.

**Recommendation: Option A.** The job's "when cheap to compute" guard rules out C, and B's UX cost outweighs its informational value for typical messages. Lock in `Starting on it…` and revisit only if operators request more.

### 3. Should the affordance be applied to the `messageHandler` delegation branch (lines 121–123) too, or only the in-adapter Claude path?

No production code currently calls `channel.onMessage(...)` (verified via grep). Future extensions might (e.g. an external orchestrator).

- **Option A — Apply to both branches** by wrapping the entire post-routing block in a `withTypingIndicator(ctx, async () => …)` helper. Future-proof.
- **Option B — Apply only to the in-adapter Claude path.** Smaller diff. External handlers, if added later, can opt-in themselves.

**Recommendation: Option B.** Keeps the change small and obviously correct. External handlers that want the affordance later can call a small exported helper, but we shouldn't impose typing-loop semantics on hypothetical callers we don't have.

### 4. Refresh interval value?

Telegram clears the indicator after ~5s; the job suggests "every 4 seconds." 4000ms is the conventional value (1s safety margin). Confirming so the planning agent doesn't have to re-derive it.

**Recommendation: `TYPING_REFRESH_MS = 4000`** with a comment citing Telegram's ~5s indicator TTL.

## PLAN

### Summary

After a configurable delay (`LONG_RUNNING_DELAY_MS = 1500`), the bot sends a fixed acknowledgement reply (`Starting on it…`) and starts a self-refreshing Telegram `typing` chat-action loop (`TYPING_REFRESH_MS = 4000`) that runs until the Claude `complete()` call resolves or throws. Cleanup is idempotent and runs from a `finally` block, so a thrown error or the existing user-visible `Sorry, …` reply path remains intact and never leaks a timer. All Telegram API calls used by the affordance are individually try/caught so a transient network error logs and degrades the affordance rather than killing the user-visible work. Scope is the in-adapter Claude path only — the unused `messageHandler` delegation branch is left untouched per operator answer Q3.

## ARCHITECTURE

### Why an "optimistic delay" timer instead of always-on?

Operator answer Q1 chose Option B. The literal reading of the requirement is "do NOT send an acknowledgement for short / instant replies." Today every Claude-routed message is long, so always-on (Option A) and optimistic-delay (Option B) are observably equivalent. But Option B is cheap (~10 lines of `setTimeout` / `clearTimeout`) and survives a future change such as adding a `/help` shortcut or a fast-path keyword router that resolves before 1.5s. Choosing A and retrofitting later would mean re-threading timer semantics through the handler. Choosing B costs almost nothing today and is robust to that change.

### Why a `finally`-driven cleanup closure rather than an explicit `withTypingIndicator` wrapper?

The existing `try/catch` already has a meaningful catch path (the user-visible `Sorry, …` reply). Wrapping the whole block in a `withTypingIndicator(ctx, async () => { … })` higher-order helper would either swallow that catch or push it inside the callback awkwardly. A cleanup closure pattern (`const stop = start(); try { … } finally { stop(); }`) preserves the existing error-handling shape exactly, requires zero refactoring of the post-complete logic (history push, chunking, storeConversation), and is easy to read in-place. It's also the standard Node pattern for paired interval/timeout resources.

### Why log-and-degrade on `sendChatAction` / ack-reply failures?

The requirement is explicit: "Token / chat ID / network errors from the Telegram API must not crash the orchestration task; they should be logged and the task continues without the affordance rather than failing the user-visible work." A typing-indicator failure is purely cosmetic. The user-visible work (the actual `complete()` call and final reply) is the contract; the affordance must defer to it. Each Telegram API call inside the affordance gets its own try/catch with `logger.warn` so partial failures are logged but never propagate. The refresh `setInterval` callback uses `.catch()` (not `await`) for the same reason — an unhandled rejection in a timer callback would crash the process under Node 18+'s `unhandledRejection: throw` default, which is exactly the scenario the requirement forbids.

### Why three named constants instead of one?

The requirement says "the threshold for long-running must be a single, named decision." `LONG_RUNNING_DELAY_MS` is *the* threshold and satisfies that line literally. `TYPING_REFRESH_MS` is a separate concern (a Telegram protocol detail, not a "long-running" decision). `STARTING_ACK` is the fixed wording (a UX decision per Q2). Bundling them into one constant would be misleading — the threshold value isn't the same kind of thing as the refresh cadence. Each gets its own named constant with its own inline-doc comment so the reasoning is local to the value.

### Why scope to the in-adapter Claude path only?

Operator answer Q3. The `messageHandler` delegation branch (`if (this.messageHandler) { … }`) currently has no production caller (verified by grep across `packages/cli/src`). Wrapping it preemptively would impose typing-loop semantics on hypothetical future callers we don't have, which would be premature generalization. If a future external orchestrator needs the affordance, the operator's explicit guidance is to expose the helper at that point — but we don't expose it now because exporting an unused API surface is also a cost. The current shape (private method, single call site) is the minimum viable change.

### Integration with existing patterns

- **Logging:** uses the existing `logger` instance and the `logger.warn(message, { error: String(err) })` shape that's already idiomatic in this file (e.g. lines 159, 161, 213).
- **Error handling style:** preserves the existing top-level `try/catch` with the `Sorry, …` reply; adds `finally` for cleanup. No new error-handling abstraction introduced.
- **Test mocking:** extends the existing `mockBotOn` / `mockSendMessage` pattern in `telegram.test.ts` with one additional `mockSendChatAction`. Uses vitest's built-in `vi.useFakeTimers()` — no new test infrastructure.
