# KF-004: Telegram Improvement

## PURPOSE

Today, when an operator sends a Telegram message that triggers a longer-running orchestration task, the bot goes silent while it works — no acknowledgement, no typing indicator, no signal that the message was received and is being acted on. This produces a confusing UX: the operator can't tell whether the message was lost, the bot is broken, or work is genuinely in progress. Surface those two missing affordances so the Telegram channel feels responsive and trustworthy.

## CURRENT_STATE

null

## REQUIREMENTS

### Acknowledgement message for long-running tasks

- When the bot determines a request will be long-running (e.g. anything routed through the worker / orchestration loop, or anything that will take more than ~3 seconds), send a short acknowledgement message to the same Telegram chat *before* starting the work.
- Acknowledgement content: human, terse, and informative. Default suggestion: `Starting on it…` plus a one-line restatement of what's being worked on when that's cheap to compute. Tune wording during planning if the agent finds a better convention in the existing codebase.
- Do NOT send an acknowledgement for short / instant replies — the bot already responds quickly enough that an extra "starting" message would be noise.
- The threshold for "long-running" must be a single, named decision (constant or config) with a comment explaining the choice, not scattered magic numbers.

### Typing indicator while working

- Use the Telegram Bot API `sendChatAction` (`action: "typing"`) to display the typing indicator in the chat for the duration of any long-running task.
- Telegram's typing action auto-expires after ~5 seconds. The implementation must refresh the action on a timer (e.g. every 4 seconds) so the indicator remains continuously visible until the work completes or fails.
- Stop the typing indicator promptly when the task finishes (success or failure) — do not leave a dangling refresh loop after the final reply is sent.
- The typing indicator must be cancellable / cleanup must run even if the task throws, is cancelled, or times out.

### Cross-cutting

- No regression for short / synchronous replies — they continue to respond as fast as they do today, with no acknowledgement and no typing indicator.
- No regression for other channels (Slack, WhatsApp, Discord, etc.) — this work is scoped to the Telegram adapter only.
- Add at least one test for each of the two behaviors (acknowledgement-emitted vs not, typing-indicator started + refreshed + stopped) using whatever Telegram-adapter test pattern already exists in the repo.
- Update inline help / docstrings on any new public function or setting introduced.
- Token / chat ID / network errors from the Telegram API must not crash the orchestration task; they should be logged and the task continues without the affordance rather than failing the user-visible work.
