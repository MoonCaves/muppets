# KF-005: V2 pipeline verification — exercise codex testing agent

## PURPOSE

After the v1 → v2 cutover (`a07cff4`), the running dispatcher kept a cached v1 graph and routed KF-004 from `in-review` directly to `done`, bypassing the new `testing` state. The dispatcher has been restarted and now reads the v2 graph fresh. This job exists to confirm a fresh dispatcher actually routes a new job through `dev → in-review → testing → done`, exercising the codex/openai testing agent for the first time in this module's history.

There is no code to write — the goal is to let a fresh-dispatcher v2 graph route this job through `dev → in-review → testing → done`, producing observable evidence (an `in-review → testing` transition in ACTIVITY_LOG and a TESTS section authored by the codex/openai agent) that the new `testing` state actually fires. ## PLAN_QUESTIONS

### Q1: How should KF-005 handle the red-at-HEAD test suite?

The PLAN's assumption #4 ("`pnpm test` is green at HEAD") is **false**. Dev confirmed: `pnpm test:run` reports 2 failed / 529 passed across 37 files at the worktree's branch base. Both failures are **unrelated to KF-005's purpose**:

1. `packages/cli/src/runtime/identity-watcher.test.ts > watchIdentity > fires onError on invalid YAML and keeps watching` — 5000ms timeout. The test polls a YAML watcher every 1s and waits 4.5s for a debounce; it appears to be a flaky timing test, not a behavior regression.
2. `packages/cli/src/server/web-api.test.ts > GET /memory/:block > should return 404 when file not found` — got 200, expected 404. Real behavior/test drift; needs investigation to know which side is wrong.

The v2 graph (verified at `delamain.ts` lines 100–139) has **no type-based skip** — the `testing` state fires for every job class including `chore`. So these failures will be encountered by the testing agent regardless of what dev does.

**Why this matters now:** the original PURPOSE asks for routing through `dev → in-review → testing → done`. With a red suite, the testing agent will route `testing → test-input` instead, halting at an operator gate — violating REQUIREMENTS bullet 1 ("without operator-gate halts").

**Options:**

**A. Repair the two failing tests inside KF-005's scope.**
- Pros: Self-contained — one job, one merge, smoke test passes the way PURPOSE wrote it.
- Cons: Pollutes the smoke test. Type would shift from `chore` to `bug`/`task`, and the testing agent now has real implementation deltas to verify, **coupling** the variables under test ("did the v2 graph route correctly" + "did the test fixes work"). The whole point of using a chore was to isolate the routing variable. Also: failure #1 is a flaky timer, which is hard to "fix" deterministically without rewriting the watcher's polling cadence.

**B. Spawn a separate prep job (KF-006) to repair the two tests, then re-run KF-005 after KF-006 closes.**
- Pros: Architecturally cleanest. KF-005 stays a pure observation chore. KF-006 carries the actual code change, and goes through the same v2 pipeline (which itself is more smoke-test signal). Clear separation of concerns.
- Cons: Blocks KF-005 until KF-006 lands. Two jobs in flight where one was planned. KF-006 may itself surface further unrelated failures or ambiguity, snowballing.

**C. Accept `testing → test-input` as the outcome and relax REQUIREMENTS bullet 1.**
- Pros: KF-005 runs immediately. **More v2 coverage, not less:** routing to `test-input` exercises the FAIL branch of the testing agent AND touches the `test-input` operator-gate state — both v2-only paths. The openai/codex provider still gets invoked for the first time (PURPOSE's secondary goal). PURPOSE's primary goal — "confirm a fresh dispatcher actually routes a new job through `dev → in-review → testing`" — is fully met when we observe the `testing` ACTIVITY_LOG entry and the gpt-5.5-authored TESTS section.
- Cons: PURPOSE explicitly names `→ done` as the target terminal state. C does not reach `done` automatically; the operator manually closes the job from `test-input` (via `exit: test-input → done` after acknowledging the unrelated failures, or via `rework: test-input → dev` if they want fixes — which loops back to A or B from inside the pipeline).

**Recommendation: C.**

Reasoning: the primary risk this smoke test was created to detect — "is the dispatcher still serving cached v1?" — is observable from a single signal: does `testing` appear in ACTIVITY_LOG. That signal fires under C just as well as under the original A. The secondary goal — "exercise codex/openai for the first time" — also fires under C, and arguably *more thoroughly* because the testing agent has real failures to characterize, exercising its FAIL routing logic. Option C also reveals the openai provider's behavior on a non-trivial path (writing a real failure narrative to TESTS) instead of the trivial PASS path. The cost is one operator action (close from `test-input`), which is cheap. Options A and B both spend more effort and add coupling or blocking that the smoke test was specifically designed to avoid.

If C is chosen, the PLAN edits are minimal: relax REQUIREMENTS bullet 1 to allow `testing → test-input`, update Test Plan so dev no longer gates on green-at-HEAD (just runs the suite to capture baseline state and advances regardless), and add an ACTIVITY_LOG-visible expectation that the operator will close from `test-input` after observing the smoke-test signals.

If B is chosen, this job goes to `shelved` (or stays in `planning` paused), KF-006 is created with type `bug` to repair the two failures, and KF-005 is re-opened after.

If A is chosen, the type changes from `chore` to `bug`, REQUIREMENTS gains two bullets ("identity-watcher test passes deterministically", "GET /memory/:block returns 404 on missing block"), and the smoke-test framing in PURPOSE is amended to acknowledge the coupling.

## ARCHITECTURE

### Design rationale

This job is deliberately *not* designed as a code change. The point is to verify pipeline mechanics, and any non-trivial implementation would couple two failure modes: "did the v2 graph route correctly" and "did the implementation work." Keeping dev as a near-no-op isolates the variable under test (dispatcher graph version + new testing state + openai provider readiness).

### Key patterns

- **Observation-as-implementation.** The chore type, combined with an empty Implementation Changes list, encodes the smoke-test pattern: dev does nothing material, and the act of *successfully advancing through the graph* is the deliverable. Prior chores in this module (KF-001..KF-003 predate v2; KF-004 is the negative example) do not establish a pattern, so KF-005 is establishing one — minimal job-file mutation, no source changes, downstream agents tolerate sparse sections.
- **Vacuous test coverage.** REQUIREMENTS does not demand new tests. The testing agent's procedure (`agents/testing.md`) explicitly permits authoring zero tests when nothing testable was added. Dev's `pnpm test` is the green-at-HEAD precondition, not a new-test authoring step.
- **Gate-skip via literal `null`.** Both `RESEARCH_QUESTIONS` and `PLAN_QUESTIONS` are set to literal `null` (single-line, no other text) per the gate-routing protocol the agents use to decide whether to halt for operator input. This is the only mechanism available to enforce REQUIREMENTS' "no operator-gate halts" constraint from inside the job file itself.

### Integration with existing code

- The job depends on the v2 delamain definition at `.als/modules/kyberbot-factory/v2/delamains/kyberbot-factory-jobs/delamain.ts` being the active graph the dispatcher reads. No changes are made to that file.
- The job exercises the agent at `.als/modules/kyberbot-factory/v2/delamains/kyberbot-factory-jobs/agents/testing.md` for the first time. No changes are made to that file.
- The dev agent's commit message format (`kyberbot-factory: {id} dev → in-review`) is the existing convention from `agents/dev.md` step 5; KF-005 follows it unchanged.
