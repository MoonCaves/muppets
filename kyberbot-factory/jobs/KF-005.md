# KF-005: V2 pipeline verification — exercise codex testing agent

## PURPOSE

After the v1 → v2 cutover (`a07cff4`), the running dispatcher kept a cached v1 graph and routed KF-004 from `in-review` directly to `done`, bypassing the new `testing` state. The dispatcher has been restarted and now reads the v2 graph fresh. This job exists to confirm a fresh dispatcher actually routes a new job through `dev → in-review → testing → done`, exercising the codex/openai testing agent for the first time in this module's history.

## RESEARCH

**Feasibility: green.** Job is a chore/observation smoke test; the only "implementation" is letting the pipeline run and observing the artifacts produced.

**What needs to be true for a pass:**
1. Fresh dispatcher reads `.als/modules/kyberbot-factory/v2/...` (not cached v1). PURPOSE asserts the restart already happened — observable downstream as `testing` appearing in ACTIVITY_LOG.
2. Each gate (`research-gate`, `planning-gate`) routes `advance` (not to `*-input`) so the operator is never paused. Risk noted below.
3. The `in-review` agent issues an `advance: in-review → testing` rather than the (now-defunct) direct-to-done path KF-004 took.
4. The `testing` agent (gpt-5.5 / openai) successfully spawns, reads REQUIREMENTS, finds nothing testable (chore, no new code), writes a TESTS section, and routes `exit: testing → done`. The agent prompt at `agents/testing.md` allows authoring no tests so long as the suite runs and passes; for a chore with no implementation deltas, "all REQUIREMENTS-implied behavior is covered" is vacuously true.

**Identified risks / things to watch (not blockers):**
- *Operator-gate halts.* REQUIREMENTS forbid them. The research-gate and planning-gate agents may still elect to route to `research-input` / `plan-input` if they consider this chore underspecified. Setting `RESEARCH_QUESTIONS` to literal `null` here is the explicit signal that downstream `research-gate` should skip the operator gate (per agent procedure step 5). Same protocol must hold at planning-gate.
- *Test command discovery.* Testing agent procedure step 5 prefers `package.json` `scripts.test`. Repo has `package.json` and `vitest.config.ts` at root — gpt-5.5 should resolve cleanly to `pnpm test` or equivalent. Suite must be green at HEAD for the pass branch to fire.
- *Codex/openai provider readiness.* This is the first job to actually invoke the openai provider. If the provider is misconfigured (API key, model alias `gpt-5.5`, sandbox/approval handshake), the dispatcher will fail before `testing.md` runs and the job will not advance. Failure mode would manifest as no `testing` ACTIVITY_LOG entry rather than a routed `test-input`.
- *Type=chore, no implementation.* The `dev` agent should produce a near-empty diff (or a no-op note). No PLAN/ARCHITECTURE/REVIEW prior art to mimic; downstream agents must tolerate sparse sections.

**Prior art:**
- KF-004 ran the v1 graph (in-review → done direct). It's the negative example — KF-005 must NOT replicate that path.
- KF-001..KF-003 predate v2; not informative for the testing-state behavior.

**No outstanding questions.** Requirements are explicit and binary-verifiable from ACTIVITY_LOG + TESTS contents at job close.
