## ARCHITECTURE

**Design decision: no-op chores are first-class.** The factory pipeline must support jobs whose purpose is verification of the pipeline itself, not domain work. A smoke test that produces a code diff is not a smoke test — it is an unrelated change wearing a smoke-test label. The architecture therefore treats "zero diff + explicit no-op declaration in ACTIVITY_LOG" as a valid dev outcome for `type: chore` jobs.

**Why this approach over alternatives:**

- *Alternative A: have planning shelve the job before dev runs.* Rejected — that would shortcut the smoke test. The point is to exercise dev and in-review agents as part of the pipeline transit. Skipping them leaves their behavior unverified.
- *Alternative B: have dev fabricate a trivial change (touch a file, bump a comment).* Rejected — fabricated changes pollute the repo, set a precedent for noise commits, and produce a false-positive smoke signal. A smoke test that lies about what the pipeline did is worse than no smoke test.
- *Alternative C (chosen): dev produces a zero-diff commit on the job file only, with an explicit no-op attestation.* This exercises the full pipeline transit while keeping the audit trail honest.

**Integration with existing pipeline:**

- Relies on the established ACTIVITY_LOG convention as the audit trail of record (each agent appends a dated entry on transition; described in CURRENT_STATE).
- Relies on the status-field-as-state-machine pattern coded in `delamain.ts` — dev advances `status: dev → in-review` as its sole state mutation, identical to how every other transition in the pipeline works.
- Reuses the planning-gate `null` PLAN_QUESTIONS convention to skip the operator gate, mirroring how research-gate already routed past `plan-input` per ACTIVITY_LOG line 89.
- No new patterns are introduced. This plan is deliberately minimal because the goal is to exercise existing patterns, not to invent new ones.

**Failure-mode posture:**

If dev's contract refuses to commit a zero-diff job, that is a *finding* of the smoke test, not a planning failure. Dev should record the refusal in ACTIVITY_LOG and route to `shelved` or back to `plan-input` for operator review. The smoke test still produced a useful signal: the pipeline cannot self-close no-op chores, and that gap should be filed as a follow-up job.
