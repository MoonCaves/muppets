# KF-003: Add fleet status CLI command

## PURPOSE

Operators need a fast way to check fleet health from the terminal ÔÇö whether the orchestration loop is running and which agents are currently active in the fleet. Today this requires reading logs or hitting an internal endpoint, which is slow and discoverable only to people who already know the codebase.

## CURRENT_STATE

A `kyberbot fleet status` subcommand already exists in `packages/cli/src/commands/fleet.ts` (lines 397ÔÇô512), wired into the top-level CLI via `createFleetCommand()` in `packages/cli/src/cli.ts:116`. KF-003 is therefore an **enhancement**, not a greenfield command.

What the existing command does today:
- Loads registered agents from `~/.kyberbot/registry.yaml` via `getRegisteredAgents()` (`packages/cli/src/registry.ts`).
- Calls the fleet-server `/health` endpoint at `http://localhost:3456/health`. Hardcoded port (`fleet.ts:426`).
- Normalizes the fleet `/health` response (which keys services as an object) into a flat `{name, status}[]` shape.
- Falls back to per-agent PID + per-agent `/health` probe for legacy independent-mode installs.
- Renders one line per agent with a status icon (`ÔŚĆ` green/yellow vs `ÔŚő` gray) and indents each running service underneath.
- Prints `X/Y agents running` summary.
- Always exits 0 (no explicit `process.exit` based on health).
- Help text: `description('Health dashboard for all agents')`.
- No `--json` option.
- "No agents registered" message when registry is empty; does not differentiate "fleet server not running" from "fleet server reachable but reports zero agents".

## REQUIREMENTS

- New CLI subcommand (e.g. `kyberbot fleet status`) that prints fleet runtime status.
- Output must include:
  - Fleet running state: `running` / `stopped` / `degraded` (or equivalent boolean + reason).
  - List of agents currently active in the fleet, with each agent's id/name and (where available) role and current state.
  - Last-tick / heartbeat timestamp if the fleet exposes one.
- Exit code: 0 when fleet is running and reachable, non-zero when stopped or unreachable ÔÇö so the command is scriptable.
- Output format: human-readable by default; support `--json` for machine-readable output.
- Handle the "fleet not configured" / "no agents" case with a clear message rather than a stack trace.
- Update CLI `--help` and any relevant docs.

### Risks / constraints

- The fleet server port is hardcoded to `3456` in `fleet.ts:426`. Multi-fleet-on-one-host is out of scope per current architecture; safe to keep hardcoded but worth flagging.
- Adding a new field (`lastBeat`) to `AgentRuntimeStatus` ripples into the `/health` response shape and any consumer that types it (desktop app, telemetry). ## RESEARCH_QUESTIONS

1. **Enhance the existing `kyberbot fleet status` subcommand, or add a new top-level command?**
   The PURPOSE in KF-003 reads naturally as "the existing surface is hard to find / underbuilt"; the path `kyberbot fleet status` already exists at `packages/cli/src/commands/fleet.ts:400`. Why it matters: this changes the diff size and discoverability story.
   - Option A ÔÇö enhance the existing subcommand in place. Smaller diff; keeps the namespace tidy.
   - Option B ÔÇö also add a top-level `kyberbot status --fleet` (or alias `kyberbot fleet:status`) so it's the first thing operators try.
   - **Recommend A** unless you specifically want a more-discoverable top-level alias. The existing `kyberbot status` covers single-agent health; keeping `fleet status` as the fleet sibling is symmetrical.

2. **Where does the "last-tick / heartbeat timestamp" come from?**
   The fleet `/health` endpoint returns its own server `timestamp` (now), but no per-agent last-beat. The closest real signal is `lastOrchTick` in `packages/cli/src/services/heartbeat.ts:149`, which is module-local and not exposed.
   - Option A ÔÇö surface `/health.timestamp` and label it "fleet last response" (cheap, but it's just the wall clock when the request hit).
   - Option B ÔÇö add `lastBeat: ISO string | null` to `AgentRuntimeStatus`, populated by the heartbeat service each tick, and render it per-agent. Most truthful, larger change (touches `agent-runtime.ts`, `fleet-manager.ts` `/health` and `/fleet`, and any typed consumer).
   - Option C ÔÇö defer the heartbeat-timestamp requirement entirely; ship A as a placeholder and file a follow-up.
   - **Recommend B** ÔÇö the PURPOSE specifically calls out heartbeat freshness as a thing operators check by reading logs today; a real per-agent `lastBeat` is the high-value datum.

3. **Exit-code contract ÔÇö how strict?**
   KF-003 says "0 when fleet is running and reachable, non-zero when stopped or unreachable." The fleet `/health` already distinguishes `'ok'` (all running) from `'degraded'` (some agent down).
   - Option A ÔÇö `0` if `/health` is reachable AND `status === 'ok'`; `1` otherwise. Strictest; fails CI on any degradation.
   - Option B ÔÇö `0` if `/health` reachable (including `'degraded'`); `1` only if unreachable/no fleet server. Matches the literal wording "running and reachable".
   - Option C ÔÇö three-state: `0` ok, `1` unreachable, `2` degraded.
   - **Recommend B** for the documented contract (matches the wording) plus a `--strict` flag that promotes `degraded` Ôćĺ exit 1, so scripted callers can opt into A. Avoid C ÔÇö non-zero-but-different exit codes are rarely consumed correctly.

4. **Should remote agents (registry `type: 'remote'`) be included?**
   The existing `fleet list` probes remotes; the existing `fleet status` ignores them. Remote agents don't run inside the local fleet manager, so they aren't in `/health.agents`.
   - Option A ÔÇö include them, probing `${remoteUrl}/health` in parallel and rendering them in a separate "Remote" group. They do NOT count toward the fleet-level verdict (different host).
   - Option B ÔÇö keep ignoring them; out of scope for "is the fleet running".
   - **Recommend A** ÔÇö operators typically think of the registry as "the fleet" regardless of host. Group them visibly and exclude from the running/degraded verdict so semantics stay clean.

5. **JSON output shape ÔÇö pass-through or normalized?**
   Two reasonable shapes for `--json`:
   - Option A ÔÇö pass through `/health` essentially as-is, plus a top-level `fleet: 'running' | 'degraded' | 'stopped'` and `agents` array. Closest to the wire; minimal translation.
   - Option B ÔÇö normalize into a flat shape: `{ fleet: { state, uptime, lastTick }, agents: [{ name, role?, state, lastBeat, services: [{name,status}] }], remote: [...] }`. Easier for scripts; stable across `/health` shape changes.
   - **Recommend B** ÔÇö the `/health` shape mixes `services` as object vs flat array between fleet and per-agent modes (already a footgun, see `fleet.ts:421-450`). A normalized CLI shape insulates scripts from that.

## ARCHITECTURE

### Design decisions and rationale

- **Why expose `lastBeat` from `heartbeat.ts` instead of tracking it in `AgentRuntime`?** The heartbeat tick function lives in `services/heartbeat.ts` and is invoked by a `setInterval` started from `AgentRuntime.start()`. The runtime never sees the tick callback fire ÔÇö only the timer handle. Pushing the timestamp through the runtime would require a callback wire (constructor option or event emitter). Pulling it via `getLastBeat(root)` is one line and matches the existing `lastOrchTick` precedent in the same file.
- **Why a per-root Map instead of a per-agent field?** The heartbeat's root keying is already established (`tick(root)`, identity loaded by root). Switching to `agent_name` keying would force a name lookup inside `tick()` and create a second source of truth. Roots are unique within a fleet by construction (port collisions are prevented by `getNextAvailablePort`), so the Map is safe.
- **Why kill the legacy per-agent independent-mode fallback?** The current code carries two probe paths (fleet `/health` AND per-agent PID + port). With the operator confirming fleet-first (Q1), the second path is dead weight that misreports running fleet-mode agents as `starting (PID ÔÇŽ)` when fleet-server logic skips them. Cleaning it up reduces conditional branches in the renderer.
