# KF-003: Add fleet status CLI command

## PURPOSE

Operators need a fast way to check fleet health from the terminal ŌĆö whether the orchestration loop is running and which agents are currently active in the fleet. Today this requires reading logs or hitting an internal endpoint, which is slow and discoverable only to people who already know the codebase.

## CURRENT_STATE

A `kyberbot fleet status` subcommand already exists in `packages/cli/src/commands/fleet.ts` (lines 397ŌĆō512), wired into the top-level CLI via `createFleetCommand()` in `packages/cli/src/cli.ts:116`. KF-003 is therefore an **enhancement**, not a greenfield command.

What the existing command does today:
- Loads registered agents from `~/.kyberbot/registry.yaml` via `getRegisteredAgents()` (`packages/cli/src/registry.ts`).
- Calls the fleet-server `/health` endpoint at `http://localhost:3456/health`. Hardcoded port (`fleet.ts:426`).
- Normalizes the fleet `/health` response (which keys services as an object) into a flat `{name, status}[]` shape.
- Falls back to per-agent PID + per-agent `/health` probe for legacy independent-mode installs.
- Renders one line per agent with a status icon (`ŌŚÅ` green/yellow vs `ŌŚŗ` gray) and indents each running service underneath.
- Prints `X/Y agents running` summary.
- Always exits 0 (no explicit `process.exit` based on health).
- Help text: `description('Health dashboard for all agents')`.
- No `--json` option.
- "No agents registered" message when registry is empty; does not differentiate "fleet server not running" from "fleet server reachable but reports zero agents".

## REQUIREMENTS

- New CLI subcommand (e.g. `kyberbot fleet status`) that prints fleet runtime status.
- Output must include:
  - Fleet running state: `running` / `stopped` / `degraded` (or equivalent boolean + reason).
  - List of agents currently active in the fleet, with each agent's id/name and (where available) role and current state.
  - Last-tick / heartbeat timestamp if the fleet exposes one.
- Exit code: 0 when fleet is running and reachable, non-zero when stopped or unreachable ŌĆö so the command is scriptable.
- Output format: human-readable by default; support `--json` for machine-readable output.
- Handle the "fleet not configured" / "no agents" case with a clear message rather than a stack trace.
- Update CLI `--help` and any relevant docs.

### Risks / constraints

- The fleet server port is hardcoded to `3456` in `fleet.ts:426`. Multi-fleet-on-one-host is out of scope per current architecture; safe to keep hardcoded but worth flagging.
- Adding a new field (`lastBeat`) to `AgentRuntimeStatus` ripples into the `/health` response shape and any consumer that types it (desktop app, telemetry). Search points: `getStatus()` at `agent-runtime.ts:248`, `getAllStatuses()` at `fleet-manager.ts:596`, both `/health` (lines 129, 361) and `/fleet` (line 153) responses.
- Remote agents in the registry probe a different host (`entry.remoteUrl`); aggregating them into the same fleet-level "running" verdict requires defining what "fleet running" means when half the agents are remote. Recommend treating remote-agent reachability as a separate column rather than folding it into the fleet verdict.
- `displayServiceStatus` from `splash.ts` is reused in `commands/status.ts` ŌĆö could be reused here, but the existing fleet rendering is already custom; not a blocker either way.

## RESEARCH_QUESTIONS

1. **Enhance the existing `kyberbot fleet status` subcommand, or add a new top-level command?**
   The PURPOSE in KF-003 reads naturally as "the existing surface is hard to find / underbuilt"; the path `kyberbot fleet status` already exists at `packages/cli/src/commands/fleet.ts:400`. Why it matters: this changes the diff size and discoverability story.
   - Option A ŌĆö enhance the existing subcommand in place. Smaller diff; keeps the namespace tidy.
   - Option B ŌĆö also add a top-level `kyberbot status --fleet` (or alias `kyberbot fleet:status`) so it's the first thing operators try.
   - **Recommend A** unless you specifically want a more-discoverable top-level alias. The existing `kyberbot status` covers single-agent health; keeping `fleet status` as the fleet sibling is symmetrical.

2. **Where does the "last-tick / heartbeat timestamp" come from?**
   The fleet `/health` endpoint returns its own server `timestamp` (now), but no per-agent last-beat. The closest real signal is `lastOrchTick` in `packages/cli/src/services/heartbeat.ts:149`, which is module-local and not exposed.
   - Option A ŌĆö surface `/health.timestamp` and label it "fleet last response" (cheap, but it's just the wall clock when the request hit).
   - Option B ŌĆö add `lastBeat: ISO string | null` to `AgentRuntimeStatus`, populated by the heartbeat service each tick, and render it per-agent. Most truthful, larger change (touches `agent-runtime.ts`, `fleet-manager.ts` `/health` and `/fleet`, and any typed consumer).
   - Option C ŌĆö defer the heartbeat-timestamp requirement entirely; ship A as a placeholder and file a follow-up.
   - **Recommend B** ŌĆö the PURPOSE specifically calls out heartbeat freshness as a thing operators check by reading logs today; a real per-agent `lastBeat` is the high-value datum.

3. **Exit-code contract ŌĆö how strict?**
   KF-003 says "0 when fleet is running and reachable, non-zero when stopped or unreachable." The fleet `/health` already distinguishes `'ok'` (all running) from `'degraded'` (some agent down).
   - Option A ŌĆö `0` if `/health` is reachable AND `status === 'ok'`; `1` otherwise. Strictest; fails CI on any degradation.
   - Option B ŌĆö `0` if `/health` reachable (including `'degraded'`); `1` only if unreachable/no fleet server. Matches the literal wording "running and reachable".
   - Option C ŌĆö three-state: `0` ok, `1` unreachable, `2` degraded.
   - **Recommend B** for the documented contract (matches the wording) plus a `--strict` flag that promotes `degraded` ŌåÆ exit 1, so scripted callers can opt into A. Avoid C ŌĆö non-zero-but-different exit codes are rarely consumed correctly.

4. **Should remote agents (registry `type: 'remote'`) be included?**
   The existing `fleet list` probes remotes; the existing `fleet status` ignores them. Remote agents don't run inside the local fleet manager, so they aren't in `/health.agents`.
   - Option A ŌĆö include them, probing `${remoteUrl}/health` in parallel and rendering them in a separate "Remote" group. They do NOT count toward the fleet-level verdict (different host).
   - Option B ŌĆö keep ignoring them; out of scope for "is the fleet running".
   - **Recommend A** ŌĆö operators typically think of the registry as "the fleet" regardless of host. Group them visibly and exclude from the running/degraded verdict so semantics stay clean.

5. **JSON output shape ŌĆö pass-through or normalized?**
   Two reasonable shapes for `--json`:
   - Option A ŌĆö pass through `/health` essentially as-is, plus a top-level `fleet: 'running' | 'degraded' | 'stopped'` and `agents` array. Closest to the wire; minimal translation.
   - Option B ŌĆö normalize into a flat shape: `{ fleet: { state, uptime, lastTick }, agents: [{ name, role?, state, lastBeat, services: [{name,status}] }], remote: [...] }`. Easier for scripts; stable across `/health` shape changes.
   - **Recommend B** ŌĆö the `/health` shape mixes `services` as object vs flat array between fleet and per-agent modes (already a footgun, see `fleet.ts:421-450`). A normalized CLI shape insulates scripts from that.
